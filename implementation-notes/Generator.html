scaffold-optimize.cjs (the loop-conductor generator)

Config is data-in via args, NOT baked into the template

The biggest structural deviation from conductor's scaffold-workflow.cjs: conductor bakes the per-phase prompts into the generated template literal (escaped). Prospector does the opposite — the generator emits ONLY the loop structure and the loop reads goal/surface/metric/gate/budgets from args.config at runtime. This was a hard requirement from the spec/SKILL ("passed via Workflow args at launch (NOT embedded), so resume re-passes it"). The generated generate(name) therefore takes only the slug; no per-config regeneration is needed, and a resume re-passes the same config unchanged. Consequence: the generator's CLI lost all of conductor's flag/profile/prompt machinery — there is nothing project-specific to bake.

Helpers inlined via Function.prototype.toString() to guarantee no drift

The three pure decision helpers (isBetter / shouldStop / deriveEvalCap) must exist in TWO places: exported from the generator module (so test-optimize.cjs can unit-test them) AND inlined verbatim into the generated conductor (the restricted layer can't require them). Rather than maintain two copies, I build HELPERS_SRC by calling fn.toString() on the real exported functions and splicing the source into the template. The inlined copy can never drift from the tested one. Tradeoff: the inlined helpers must stay self-contained (no closure over module-scope vars) — they are, by design (pure scalar in/out).

surfaceAllows (surface lock) is a hand-rolled glob matcher, conductor-side, not exported

The surface lock (spec §4) must run in the restricted conductor — it decides keep/discard from the worker-reported diffFiles, and the conductor can't shell out to git/minimatch. So I inlined a minimal glob→RegExp matcher supporting ** (crosses /), * (not /), and ?. Decisions:

  • Empty diff ⇒ not kept. surfaceAllows([], …) returns false — a "change" that touched zero files has nothing to commit and shouldn't be recorded as KEPT. The eval worker reporting an empty git diff --name-only means the propose edit was a no-op.
  • Multiple patterns via comma/newline split (a surface may be a list) — a file passes if it matches ANY pattern; the experiment is kept only if EVERY file passes.
  • It is NOT exported from the generator module (it's generated as a string with escaped regex); the test net extracts and exercises it from the emitted file instead. Verified against within/outside/deep/multi-pattern/empty/leading-./ cases.

Resume must NOT re-measure the baseline — but still needs an eval cap

On a fresh run the Baseline phase measures evalSec and derives evalCapSec ≈ 3×evalSec. On resume, best arrives via args and re-running the baseline would (a) waste an eval and (b) risk overwriting best from a noisy re-measurement. So the generated loop skips the Baseline phase entirely when best != null. Open consequence: a resumed run then has no freshly-measured evalSec to derive the cap from. I resolved this by: prefer a pinned budgets.evalCapSec; else use an optional args.evalSec the main session can re-pass; else fall back to the 30s floor via deriveEvalCap(0). A reviewer should confirm the SKILL's resume step re-passes evalSec (or that pinning evalCapSec for resumable runs is acceptable) — otherwise a resumed run uses the floor cap, which is tighter than the fresh run's derived cap.

Stop counters the conductor can't read (wallclock, tokens) come from args

shouldStop takes elapsedMin/tokensUsed as parameters because the conductor has no clock and no token meter. Within a single live run, only the experiments count and plateau stops fire (both derivable from pure counters). The wallclock/token stops are honoured at resume time (the main session passes the measured-so-far values via args.elapsedMin/args.tokensUsed). This matches the spec's "the conductor cannot write timestamps; push that into workers" — here the stop signal is pushed in via resume rather than computed live. A reviewer may want per-experiment in-loop wallclock enforcement; that would require the checkpoint worker to feed elapsed time back, which the current one-way checkpoint does not do (deferred — matches §3's emphasis on the per-experiment caps as the real bound, with totals as a coarse outer stop).

Discard reverts with git clean -fd inside the surface

git checkout -- <surface> alone does not remove NEW untracked files an experiment added inside the surface glob (the spec explicitly allows new matching files). So the revert worker also runs git clean -fd -- <surface> and asserts git status --porcelain is empty. Scoped to the surface so it never touches node_modules symlink / state files elsewhere.
