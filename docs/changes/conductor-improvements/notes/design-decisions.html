Implementation notes

conductor · model-mode + Writeup · 2026-06-27

The non-trivial decisions a reviewer should confirm. (This change was hand-built, so these are the author's notes — exactly the kind of fragment the Writeup phase promotes from a real worker run.)

decisionByte-identity is the load-bearing invariant

Every new fragment (the model: opt, the Writeup phase, the PR-body artifact line) is emitted only when opted in. The proof: the pre-change generator and the new one with --no-writeup in default mode produce identical bytes across the whole flag matrix, except the one intentional DocsGate summary.md path line. This is why the emitters compose [agentFrag, modelFrag] and fall back to the exact prior string when both are empty — a trailing space would have broken it.

decisionPhase → model-class taxonomy

mechanical=haiku (Setup, checkpoint, Verify, ReVerify, PR, TicketUpdate), think=opus (Brief, RED, PathGap, CodeGate, Review, TestGate, DocsGate, Writeup), work=sonnet. Two judgement calls worth a second opinion: Brief is "think" because it derives the goal that gates the whole run; Verify/ReVerify are "mechanical" because they run a known suite and report pass/fail. Each is a one-word change on the descriptor.

tradeoffWriteup hard-fails before PR

Placing Writeup before PR and making it throw on failure means "no artifacts ⇒ no PR" — which is the strict reading of "every PR shall have a write-up." The cost: a writeup that genuinely can't be produced blocks the PR. Mitigation: --no-writeup is the escape hatch, and the run is resumable, so a fix + resume re-runs only Writeup onward. The alternative (soft, non-blocking) was rejected as not honoring the requirement.

tradeoffSibling skills invoked by name, not by path

conductor runs on any repo, but pr-writeup/flowchart live in the plugin, not the target tree. The Writeup worker invokes them as lirbox:pr-writeup / lirbox:flowchart (portable wherever the plugin is installed) with a read-the-SKILL.md fallback. The deterministic eval can only assert the prompt wiring; the actual generation quality is a runtime property.

open questionSequence diagrams

The ask mentioned "flowchart, sequence, etc." v1 emits exactly one design diagram and lets the worker pick the Mermaid type (flowchart or sequenceDiagram) that fits. If multi-diagram output is wanted later, that is a follow-up — not in this change.
